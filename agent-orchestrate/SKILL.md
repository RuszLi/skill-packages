# Agent Orchestrate

## 概述

使用此技能时，主对话只做编排，不做执行。具体工作必须交给独立上下文 subagent；主对话只负责读取项目规则、拆分任务、选择模型、打包上下文、分派、审查结果、冲突判断、核验证据和向用户报告。

## 平台适配

本技能默认使用“通用编排语义”，并在不同平台落到各自原生工具链。

- **在 Claude Code 中：** 使用 `Agent` 工具分派 subagent；需要立即结果时使用前台 `Agent`，独立长任务才用 `run_in_background: true`。
- **在 Codex 中：** 使用 `spawn_agent` / `wait_agent` / `close_agent` / `resume_agent` 等原生多智能体工具。
- **在其他平台中：** 使用该平台的原生 subagent / teammate / task 工具，但必须满足本技能的同一组约束：自包含任务包、真实等待、结果审查、证据核验。

技能正文优先写“编排语义”，不是绑定单个平台的工具名。需要落地时：

- Claude Code 参考 `references/claude-code-tools.md`
- Codex 参考 `references/codex-tools.md`
- 其他平台参考各自的 tools 映射文档

## 硬规则

- 主对话只做编排，不做执行。
- 「执行 = 实现、修复、重构、诊断、审计、批量搜索、验证命令、文件编辑」；这些事项必须交给 subagent。
- 主会话不得直接实现、修复、重构、诊断、审计、批量搜索、运行验证命令或编辑业务文件。
- 开发、排查、验证、审查、计划执行等具体事项必须交给 subagent。
- 每个 subagent 必须收到独立、自包含的任务包。
- 不得依赖主会话历史、隐藏推理或“见上文”。
- 主会话只负责读取项目规则、拆分任务、选择模型、打包上下文、分派、审查结果、冲突判断、核验证据和向用户报告。

### 规则加载最小化

主会话负责完整读取项目规则文件。

subagent 默认不重复全量加载项目中的 `AGENTS.md`、`SOUL.md`、`USER.md`、`CLAUDE.md` 或同类规则文件，除非任务风险或复杂度要求这样做。

默认做法：

- 主会话读取完整规则。
- 主会话将与当前任务直接相关的最小必要约束压缩成摘要。
- subagent 只接收该摘要，而不是重复接收整套规则文件。

默认摘要至少应包含：

- 当前任务的成功标准与禁止事项。
- 项目级硬约束，例如安全边界、未来数据禁令、关键验证底线。
- 与该任务直接相关的用户协作偏好。
- 允许修改和禁止修改的文件范围。

仅在以下情况允许向 subagent 传递完整规则文件或其大段原文：

- 复杂根因分析。
- 架构级、跨模块、跨阶段改动。
- 高风险安全、交易、权限、资金、审计链路任务。
- 主会话判断若无全文约束，subagent 容易越界或误判。

轻量搜索、字段核对、局部实现、机械性修改、常规测试建议等任务，默认只传规则摘要。

## 平台 continuation 契约

### Claude Code

Claude Code 中，subagent 返回必须显式落到 `Agent` 工具链：

`Agent -> 审查返回 -> 继续分派或结束`

说明：此处“继续分派”可为并行继续分派；满足并行条件时应保持并行优先，不表示默认顺序串行。

硬约束：

- 分派 subagent 后，如果后续决策依赖其结果，必须使用前台 `Agent`；不能用自然语言假装“等待它回来”。
- 若使用 `run_in_background: true`，只能用于独立长任务；主会话继续处理别的工作，等待平台完成通知。
- 前台 `Agent` 返回后，主会话必须立即审查返回内容，并根据结果进入“继续分派、要求修正、最终报告”三选一。
- 后台 `Agent` 完成后，主会话也必须审查返回内容；不能因为收到通知就直接宣称完成。
- 禁止用 `sleep`、轮询或自然语言承诺代替平台真实等待机制。

### Codex

Codex 中，subagent 返回必须显式落到 multi-agent 工具链：

`并行优先批量 spawn_agent -> wait_agent(任一完成或批量完成) -> 审查返回 -> 同链缺口用 send_input 或 resume_agent 后继续 wait -> 任务链结束后 close_agent`

硬约束：

- 分派 subagent 后，主会话不得直接结束本轮；必须调用 `wait_agent` 等待该 agent 的最终状态，除非用户明确要求后台运行。
- 可独立推进且写入范围不冲突的任务，默认一次性并行 `spawn_agent`；只有存在结果依赖、写集冲突或集成前置验收时才串行。
- 并行分派后，主会话应使用 `wait_agent` 等待任一完成或批量完成；未完成且仍有后续价值的 agent 保持 open，不提前 `close_agent`。
- `wait_agent` 返回 completed 状态后，主会话必须立即审查返回内容，并根据结果进入“继续分派、要求修正、最终报告”三选一。
- `close_agent` 只在任务链结束、结果已接受、无需继续该上下文时调用；不得长期占用不需要的 agent。
- 有缺口时先判断是否属于同一任务链；同链优先继续原 agent（`send_input` 或 `resume_agent` 后 `wait_agent`），仅在独立任务或上下文不相关时新分派 agent。
- 若 `wait_agent` 超时，主会话只能做状态更新，不能宣称完成；下一动作必须是继续 `wait_agent`、发送补充输入或向用户报告明确阻塞。
- 禁止把“等待子 agent 回复”写成自然语言承诺；等待必须对应实际 `wait_agent` 调用。

## Subagent 任务包

每个 subagent 提示词必须包含：

1. 任务目标和成功标准。
2. 任务分类声明：当前任务属于哪种任务类型，必须使用哪个模型和推理等级，或在 Claude Code 中必须使用哪个 `subagent_type` / `model`。
3. 范围边界，以及涉及的文件或子系统。
4. 必要上下文：规格、错误、日志、相关路径、约束和安全规则。
5. 允许修改和禁止修改的内容。
6. 必须运行的验证命令或必须提供的证据。
7. 预期输出：根因、修改内容、测试结果、剩余风险。

关于规则打包，必须遵守：

- 优先传“任务相关规则摘要”，不要默认附带完整项目规则全文。
- 摘要应写成可执行约束，而不是仅列文件名。
- 若确实附带完整规则文件，必须说明为什么摘要不足。
- 同一批并行 subagent 若共享同一套摘要，保持措辞稳定，以提高缓存复用概率。

推荐的规则摘要模板：

`项目约束摘要：<3 到 10 条与任务直接相关的硬约束>`

禁止写这类提示：

- “继续上面的任务。”
- “按我们刚才讨论的做。”
- “修复这个问题”，但不提供文件、症状和验收证据。

## 模型选择

模型选择优先级：

1. 用户当前消息明确指定的模型、推理等级或分派方式。
2. 当前项目 `AGENTS.md` / `CLAUDE.md` / 同类规则文件中的模型分配规则。
3. 默认继承父会话模型；只有用户指定或任务规则要求时才覆盖模型。
4. 若前 3 者都没有，自动判断任务类型并使用平台默认映射。

效率规则：

- 先用能满足正确性的最低成本模型，不得把所有任务默认派给最高模型。
- 低风险、结构化、可自动验收任务优先用便宜模型。
- 常规代码、修复、验证、审查任务优先用标准模型。
- 高风险、低容错、强不确定任务再升级到更强模型。
- 首轮未通过校验时，先升模型，再升推理强度。

### Codex 默认映射

| 任务类型 | 模型 | 推理等级 |
| --- | --- | --- |
| 轻量审计、日志抽取、字段核对 | `gpt-5.4-mini` | `low` |
| 纯代码、实现、机械性修改 | `gpt-5.3-codex` | `medium` |
| 常规代码审计、安全边界检查、测试覆盖审查 | `gpt-5.3-codex` | `medium` |
| 一般推理、审查、集成判断 | `gpt-5.4` | `medium` |
| 整理、小修改、摘要 | `gpt-5.4-mini` | `low` |
| 交易安全、RiskGate、权限、止损、审计链路等高风险审计 | `gpt-5.5` | `medium` 或 `high` |
| 复杂推理、架构、根因分析 | `gpt-5.5` | `high` |
| 架构级、跨系统、难以自动验收的审计 | `gpt-5.5` | `high` |

### Claude Code 默认映射

| 任务类型 | `subagent_type` | 模型 |
| --- | --- | --- |
| 轻量整理、固定格式摘要 | `general-purpose` | `haiku` |
| 纯代码、实现、机械性修改 | `general-purpose` | `sonnet` |
| 常规代码审计、安全边界检查、测试覆盖审查 | `general-purpose` | `sonnet` |
| 一般推理、审查、集成判断 | `general-purpose` | `sonnet` |
| 广泛只读搜索、批量定位、引用收集 | `Explore` | 默认或 `sonnet` |
| 复杂推理、架构、根因分析 | `Plan` 或 `general-purpose` | `opus` |
| Claude Code / Claude API / hooks / slash commands 问题 | `claude-code-guide` | 默认 |

分类 subagent 任务时，必须明文写出：

- **Codex：** `任务类型：<类型>；必须使用模型：<模型>；推理等级：<等级>；选择原因：<一句话原因>`
- **Claude Code：** `任务类型：<类型>；subagent_type：<类型>；model：<默认或显式值>；选择原因：<一句话原因>`

若任务混合多个类型，先按正确性选择满足要求的最低成本模型；只有高风险、低容错或强不确定部分主导任务时才升到更高模型。分派摘要和 subagent 任务包中都必须记录所选模型和原因。

## 工作流

1. 将用户请求拆成任务域。
2. 读取项目规则，尤其是 `AGENTS.md`、安全边界和模型映射。
3. 显式识别并标注：可并行任务、必须串行任务、共享资源、写入范围、依赖关系和集成点。
4. 拆分为 subagent 可独立理解的任务，并为每个任务声明任务类型、必须使用的模型/平台角色和选择原因。
5. 为每个 subagent 构造自包含任务包，并定义尽量 disjoint 的 ownership（文件/模块/职责/验证范围）。
6. 为每个 subagent 提炼“最小必要规则摘要”；只有高风险或高复杂度任务才附带完整规则全文。
7. 默认并行优先分派：可独立推进且写入范围不冲突的任务一次性分派；同一文件或强耦合接口的写入默认不并行，只允许并行只读审计/方案建议，最终写入串行集成或指定单一 owner。
8. 按任务类型选择模型和推理等级后，使用当前平台原生 subagent 工具分派。
9. 记录 agent id 或平台句柄、任务类型、模型、ownership、允许写入范围和预期证据。
10. 使用平台原生等待机制拿到返回；多个独立 agent 并行时，使用 `wait_agent` 等待任一完成或批量完成；对未完成或同链待续 agent 保持 open 状态。
11. 对每个 completed 返回执行审查：范围、正确性、冲突、证据缺口。
12. 平台要求显式关闭句柄时，在任务链结束、结果已接受且无需继续上下文后关闭；平台自动回收时，至少完成结果审查。
13. 有缺口时先判定是否同链：同链优先复用原 agent（`send_input` 或 `resume_agent` 后继续 `wait_agent`），独立任务或上下文不相关时再新分派；不由主会话直接修。
14. 无缺口时主会话核验 subagent 提供的验证证据并报告；需要运行最终验证时，分派独立验证 subagent。

## 技能组合

- 执行已有实现计划时，使用 `subagent-driven-development`。
- 识别到可并行且写入不冲突的任务域时，默认优先使用 `dispatching-parallel-agents`。
- 宣称完成、修复或测试通过前，使用 `verification-before-completion`，完成前必须核对新鲜验证证据；证据由 subagent 运行命令提供。
- 重要实现完成后，使用 `requesting-code-review` 做最终审查。

## 红线

以下行为属于违规：

- 主会话在本技能激活时直接编辑具体实现。
- 主会话直接执行诊断、审计、批量搜索、验证命令或文件编辑。
- subagent 提示依赖对话历史，而非自包含任务包。
- 模型选择忽略用户明确指定或项目规则的优先级。
- 模型选择无差别上最高模型，未先按风险和结构化程度做成本约束。
- 分派 subagent 时没有明文声明任务类型、模型平台角色和选择原因。
- 主会话不分派独立验证、不核验证据，或把执行型验证命令留给自己运行。
- 多个 subagent 修改同一批文件，但没有明确集成方案。
- 可并行且无冲突的任务被默认串行分派，且未说明依赖、冲突或验收前置原因。
- 任务拆分未声明 ownership（文件/模块/职责/验证范围），导致并行任务边界不清。
- 需要真实等待时却只写自然语言承诺，没有使用平台原生等待机制。
- Claude Code 中把后台 `Agent` 当成前台等待；Codex 中分派后没有实际调用 `wait_agent`。
- 只保留单个平台语义，导致另一个平台无法落地执行。

触发红线时，停止当前做法，说明违规点，并通过重新分派合格 subagent 或执行主会话验证来恢复。

## 断链恢复

如果发现 subagent 已返回但主会话没有继续：

1. 用当前平台的可恢复句柄恢复目标 agent。
2. 读取最终状态或最终返回。
3. 立即审查返回内容；若同链仍需继续，用 `resume_agent -> send_input -> wait_agent`，任务链结束后再关闭句柄。
4. 根据审查结果继续分派修正 agent 或输出最终报告。

若没有 agent id、平台句柄或可恢复状态，必须向用户说明调用链缺少可恢复句柄；不能假装已审查 subagent 结果。
